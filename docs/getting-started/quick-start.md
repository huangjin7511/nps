# 10 分钟快速开始

本页的目标是帮助你完成一条最小可用链路：

1. 启动一台 NPS 服务端
2. 连接一台 NPC 客户端
3. 建立第一条可访问的转发规则

如果你还没有完全搞清楚 NPS 是什么，只要先记住这一句：

`NPS` 在公网，`NPC` 在内网，外部访问先到 `NPS`，再由 `NPC` 转到真实服务。

如果你只想先完成第一次部署和验证，可以直接继续阅读，不需要先读概念页。

如果你完成验证后还想补充概念，再看：

- [架构与核心概念](/getting-started/architecture)
- [首次选择隧道类型](/getting-started/tunnel-types)

## 推荐第一次这样验证

第一次验证建议优先使用 TCP 隧道，而不是一开始就先配域名、HTTPS、P2P 或前置反向代理。

原因很简单：

- 不需要先准备域名
- 不需要先处理证书
- 只要内网里已经有一个 TCP 服务，就可以完成链路验证

第一次可以直接选择你已经有的一个服务：

- 有网站：先把 `127.0.0.1:8080` 这类 HTTP 服务映射到公网 TCP 端口
- 有 SSH：先把 `127.0.0.1:22` 映射到公网 TCP 端口
- 有数据库：先把对应数据库端口映射到公网 TCP 端口

建议先确认“服务端可启动、NPC 在线、外部可访问”，再继续配置域名转发和 HTTPS。这样更容易定位问题。

## 准备条件

- 一台可以从公网访问的服务器
- 一台可以访问服务端连接端口的客户端设备
- 一个已经存在的内网服务，例如 `127.0.0.1:8080` 或 `10.0.0.10:22`

## 第 1 步：部署服务端

推荐先用 Docker 完成首次验证：

```bash
docker pull duan2001/nps
docker run -d --restart=always --name nps --net=host -v <local_conf_dir>:/conf -v /etc/localtime:/etc/localtime:ro duan2001/nps
```

如果你使用发布包或源码安装，请看 [安装指南](/getting-started/install)。

## 第 2 步：确认服务端基础配置

至少确认这些配置项：

- `web_port`
- `bridge_tcp_port`
- `web_username`
- `web_password`

当前仓库自带示例配置里，常用端口通常是：

- Web 管理端：`8081`
- NPC 普通 TCP 连接：`8024`
- NPC TLS 连接：`8025`

更完整的参数说明请看 [服务端配置文件](/reference/server-config)。

## 第 3 步：访问 Web 管理端

启动后，在浏览器访问：

```text
http://<server-ip>:8081
```

仓库自带示例配置的初始账号和密码：

```text
admin / 123
```

首次登录后，先完成两件事：

1. 修改默认登录密码
2. 进入“客户端列表”，找到对应客户端并复制它的快捷启动命令

## 第 4 步：启动 NPC 客户端

最简单的方式是直接使用 Web 管理界面“客户端列表”里提供的快捷启动命令。

普通 TCP 连接示例：

```bash
./npc -server=<server-ip>:8024 -vkey=<client-vkey> -type=tcp
```

如果你使用 TLS 连接：

```bash
./npc -server=<server-ip>:8025 -vkey=<client-vkey> -type=tls
```

第一次连接时，不建议直接执行不带参数的 `npc`，也不建议先修改 `npc.conf`。
源码里的实际行为是：当 `npc` 没有拿到 `-server`、`-vkey`、`-launch` 等显式启动参数时，会回退尝试读取默认 `conf/npc.conf`。

如果你希望安装为系统服务，请看 [启动 NPC 客户端](/getting-started/start-client) 和 [客户端连接与配置](/guide/client/connect)。

## 第 5 步：先创建第一条 TCP 隧道

如果你当前的目标只是完成第一次验证，建议暂时不要先配置域名转发。

更直接的做法是先创建一条 TCP 隧道：

- 如果内网机器上有网站：把一个公网端口转到 `127.0.0.1:8080`
- 如果内网机器上有 SSH：把一个公网端口转到 `127.0.0.1:22`

在 Web 管理界面里，先找到刚才那个客户端，然后创建 TCP 隧道。

最小示例一：映射一个内网站点

- 监听端口：`10080`
- 内网目标：`127.0.0.1:8080`

完成后，外部访问：

```text
http://<server-ip>:10080
```

最小示例二：映射 SSH

- 监听端口：`10022`
- 内网目标：`127.0.0.1:22`

完成后，外部访问：

```bash
ssh -p 10022 <user>@<server-ip>
```

如果你已经确定自己就是要发布网站，再继续看域名转发；但第一次验证链路时，TCP 隧道通常更直接。

## 第 6 步：再按目标切换到正式类型

根据你的目标选择一种最常见的场景：

- 暴露内网网站：使用域名转发
- 暴露 SSH、RDP 或数据库：使用 TCP 隧道
- 暴露 DNS 或游戏端口：使用 UDP 隧道

具体配置示例请看 [隧道与转发类型](/guide/tunnels/README)。如果你还在按业务结果选方案，再看 [场景总览与选型](/guide/scenarios/common)。

## 验收结果

完成后，你至少应该能确认下面几项：

- Web 管理端可以正常登录
- 客户端状态显示为在线
- 新建的 TCP 隧道、域名转发或其他规则可以从外部访问

如果前两项正常，但最后一项不通，优先检查：

1. NPC 所在机器能否直接访问内网目标
2. 服务端防火墙是否放行了你新开的公网端口
3. 你创建的是不是适合当前服务的类型
