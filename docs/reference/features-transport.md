# 功能清单：传输与连接

这一页集中放客户端与服务端之间的传输能力、连接协议扩展和控制连接行为。

如果你要看站点、证书、Header 或 URL 路由，去看 [站点与 HTTP](/reference/features-http.md)。

## 数据压缩支持

客户端与服务端之间支持压缩传输，用于减少带宽占用并改善弱网表现。

常见开启方式：

- `npc.conf` 的 `[common]` 配置中设置 `compress=true`
- 或通过 `-launch` 的 `config.common.compress` 传入配置型启动描述

说明：

- 压缩是“客户端到服务端”链路级能力，不是单条隧道独立开关
- 新手建议先默认关闭，先完成基础连通验证后再评估是否需要开启

## 加密传输

如果外网出口会识别并限制某些业务协议，可以给客户端到服务端链路增加额外加密层。

常见方式：

- 直接使用安全连接类型：`tls` / `wss` / `quic`
- 或在 `npc.conf` 的 `[common]` 中设置 `crypt=true`

注意：

- `crypt=true` 是额外加密层，和连接类型是两个维度
- 当底层已经使用 `tls`、`quic`、`wss` 时，通常不建议再叠加 `crypt=true`
- 新部署优先先把连接类型选对，再决定是否叠加额外加密

## KCP 协议支持

在网络质量较好、链路稳定的场景，例如专线或同机房内网，可以尝试使用 KCP 作为客户端与服务端之间的连接协议。

当前配置方式是：

- 服务端在 `nps.conf` 中启用 `bridge_kcp_port`
- 客户端在命令行使用 `-type=kcp`
- 客户端在配置文件模式下使用 `conn_type=kcp`

这部分已经不再推荐使用旧的 `bridge_type` 方式。

## 多路复用

NPS 的主要通信默认已经基于多路复用，无需额外开启。

它基于 TCP 滑动窗口模型，会按延迟和带宽动态调整发送节奏。
如果长时间无法恢复，当前隧道连接会被关闭并重建。

大带宽且允许一定丢包的场景，通常先保持默认参数即可。
高并发场景可结合 [部署与兼容补充](/reference/notes.md) 里的 Linux 系统限制一起调整。

对延迟敏感且存在丢包的场景，可以按操作系统能力调整 `tcp_syn_retries`、`tcp_retries1`、`tcp_retries2` 等内核参数。

## 自定义客户端超时检测断开时间

客户端与服务端之间会周期性发送延迟测量包，这个发送周期本身不可修改。

`disconnect_timeout` 控制的是“连续多久收不到回包，就认为控制连接已经断开”。

需要注意：

- 服务端与客户端都可以设置这个值
- 如果两端设置差异很大，实际重连时机会以各自一侧的判定为准
- 客户端命令行也可以直接带 `-disconnect_timeout=60`

可在 `nps.conf`、`npc.conf` 或客户端命令行中设置 `disconnect_timeout`。
